### 通讯协议（M0 ↔ K230）

建议采用**紧凑的二进制帧协议**，每帧定长 8~16 字节，方便 M0 快速解析：

```
帧头  | 帧类型 | 数据字段... | 校验和
0xAA  | 1 byte |  N bytes    | XOR/SUM
```

**帧类型定义**：

| 类型代码 | 含义 | 数据段内容 |
|----------|------|------------|
| `0x01` | 车道线数据 | `offset`(float, mm): 横向偏差，`angle`(float, °): 航向偏差，`valid`(uint8): 是否有效 |
| `0x02` | 路口信息 | `intersection_type`(uint8): 0=无路口, 1=十字, 2=T字左, 3=T字右, 4=终点，`direction`(uint8): 应转向(0=无,1=左,2=右,3=直行) |
| `0x03` | 红绿灯状态 | `light_status`(uint8): 0=无灯, 1=红灯, 2=绿灯, 3=黄灯，`distance`(uint8): 距离灯控距离粗略值(cm) |
| `0x04` | 停车位信息 | `spot_x`(int16, mm): 空位中心x坐标(相机坐标系)，`spot_y`(int16, mm)，`spot_width`(int16, mm)，`spot_angle`(int16, °): 车位方向角 |
| `0x05` | 道闸状态 | `gate_status`(uint8): 0=未检测到道闸, 1=道闸落下(不可通行), 2=道闸抬起(可通行) |
| `0x06` | 障碍物信息 | `obstacle_x`(int16, mm): 障碍物相对坐标，`obstacle_w`(int16, mm): 宽度，`danger_level`(uint8): 0=安全, 1=注意, 2=危险 |
| `0xFF` | 心跳/状态查询 | `reserved` |

**M0 发往 K230 的指令**：
- `0x10`：开始巡线模式
- `0x11`：切换为路口检测模式
- `0x12`：切换为停车位检测模式（进入停车场后）
- `0x13`：查询红绿灯（定时发送）
- `0x14`：重置状态

这种二进制协议的好处是**解析快、占用 CPU 少、丢帧容易恢复（帧头同步）**。

## 二、核心模块设计与算法思路

### 1. 双线车道巡线（K230 视觉 → M0 控制）

**K230 端**：
- 使用 Canny 边缘检测 + 颜色分割（白色/黄色车道线）。
- ROI 取图像下半部分，透视变换为鸟瞰图。
- 在鸟瞰图中分别提取**左车道线**和**右车道线**的边缘点。
- 对左右车道线分别进行二次曲线拟合（`y = ax² + bx + c`），得到：
  - **横向偏移**：车辆中线相对于车道中心线的偏差 `offset = (左线x + 右线x)/2 - 图像中心x`（映射到实际距离 mm）。
  - **航向偏差**：左右车道线斜率的平均值 `angle = atan((b_left + b_right)/2)`。
- 输出：`offset` (mm)，`angle` (°)，`valid` (1/0)。

**M0 端**：
- 接收 `offset` 和 `angle`，使用 **PD 控制**（或 PID）计算差速量。
- 公式：`turn = Kp * offset + Kd * angle`（注意符号方向）。
- 与第一阶段巡线类似，只是偏差来源从灰度的加权平均值变成了视觉输出的物理量。

**弯道处理**：题目提到弯道时视角可降低（只看 3-10cm 范围），局部近似直线处理，因此 K230 可在弯道时缩小 ROI 范围，M0 只需降低基础速度即可。

### 2. 路口识别与自主转弯

**路口类型**：根据地图，路口有十字路口（如 A、B、E 等交叉处）和 T 字路口（地图边界处）。

**K230 端**：
- 在鸟瞰图中检测车道线的**中断特征**：当左右车道线同时向两侧扩展并消失，且前方出现横向车道线时，判定为路口。
- 区分路口类型：
  - **十字路口**：前方有横向车道线 + 左右均无墙/障碍。
  - **T 字路口（左/右）**：一侧有横向线，另一侧是边界。
- 输出：`intersection_type` + `direction`（由预设路径决定）。

**M0 端**：
- 状态机维护当前所在的路口编号（如 A→B→E...），根据**预设路径表**决定下一个动作：
  - 直行通过十字路口
  - 左转 / 右转
- 转弯控制：
  - 当检测到路口且需要转向时，M0 执行预设转弯动作：
    - **编码器里程计辅助**：记录进入路口时的编码器计数值，转过 90° 对应的轮距弧长后停止。
    - **视觉辅助**：转弯过程中 K230 持续检测车道线，当检测到新的车道线居中时，认为转弯完成。
  - 保险策略：同时用编码器 + 视觉双重判断，防止转错。

### 3. 红绿灯识别与响应

**K230 端**：
- 使用 HSV 颜色空间分割**红色**和**绿色**色块。
- 通过形状过滤（圆度、面积）排除干扰。
- 判断逻辑：
  - 如果画面中红色块面积 > 阈值 → 判定为红灯。
  - 如果绿色块面积 > 阈值 → 判定为绿灯。
- 输出：`light_status` (1=红, 2=绿) 和粗略距离。

**M0 端**：
- 当 K230 报告前方有红灯时，状态机进入 `WAIT_LIGHT` 状态，停车等待。
- 持续查询 K230，直到收到绿灯报告 → 恢复行驶。
- **计时扣除**：在状态机中维护 `total_drive_time` 和 `red_light_wait_time`，方便最后统计。

**难点**：红绿灯可能出现在路口前方，需 K230 在巡线过程中持续扫描。建议**每 200ms 查询一次**，或在路口前 30cm 开始密集扫描。

### 4. 障碍物检测与绕行

- **超声波**：用于前方近距离障碍物检测（< 50cm）。
- **K230 视觉**：可用于远距离检测（50cm ~ 1m），通过检测前方是否有非车道线的物体。
- **绕行策略**：
  - 检测到障碍物 → M0 减速并启动绕行状态机。
  - 绕行动作：左变道 → 直行超越 → 右变道回线（与第一阶段超车动作类似）。
  - 如果障碍物占据整个车道无法绕行，则停车等待（模拟等障碍物移开）。

### 5. 停车场道闸识别

**K230 端**：
- 道闸通常是一根横杆，颜色可能是红白相间或黄黑相间。
- 识别方法：
  - 颜色分割提取道闸杆的颜色区域（如红色）。
  - 检测其**角度**：横杆水平 = 落下（不可通行），横杆竖起 = 抬起（可通行）。
  - 或者检测道闸下方是否有空档。
- 输出：`gate_status`。

**M0 端**：
- 接近停车场时，切换到道闸检测模式（发指令 `0x11` 给 K230）。
- 如果道闸落下 → 停车等待，持续查询直至抬起。
- 道闸抬起后 → 通过并进入停车场。

### 6. 自主寻找空车位 & 停车

这是**最难的部分**，需要视觉 + 超声波 + 编码器的精确配合。

**找空车位**：
- K230 在停车场区域内扫描，识别**车位线框**（白色矩形框）。
- 判断框内是否有车（通过超声波检测深度变化，或视觉检测框内是否有障碍物）。
- 找到空车位后，K230 输出车位的**相对坐标（x, y）和方向角**。

**停车策略**（以侧方停车为例）：
1. 小车行驶到车位前方 20~30cm 处。
2. 执行倒车入库动作：
   - 向右打满方向倒车，直到车身与车位成 45°。
   - 回正方向继续倒车，直到车尾接近车位后线。
   - 向左打满方向前进微调，使车身完全进入车位。
3. 使用**超声波**监测后方距离，防止碰撞边界。
4. 使用**编码器**精确控制行驶距离。

**简化策略**：如果时间紧张，可以只实现**垂直倒车入库**（车位方向垂直于行驶方向），这比侧方停车简单许多，也符合题目描述。

**M0 端**：
- 添加 `Parking` 状态，内部包含子状态：`FIND_SPOT → ALIGN → BACK_IN → ADJUST → PARKED`。
- 每个子状态执行固定的动作序列，并通过视觉反馈修正。

### 7. 路线显示

- 在 M0 端维护一个**位置估算器**（里程计 + 视觉校正）。
- 路线显示：可以在 OLED 上**简易绘制**（像素级），或者通过**蓝牙发送到上位机**（PC 端 Python 脚本实时绘图）。
- 建议使用上位机方案，因为 OLED 分辨率太低无法显示清晰地图。

### 8. 行驶计时

- M0 端使用 FreeRTOS 计时器（`BspDwt_GetTimeline_Sec()`），记录：
  - `total_drive_time`：总行驶时间。
  - `red_light_wait_time`：累计等红灯时间。
  - 最终显示：`effective_time = total_drive_time - red_light_wait_time`。

- 透视变换矩阵 `M` 需要**离线标定**：在地面上放置标定板，手动选取四个点计算单应性矩阵。

## 七、总结：第二阶段核心要点

1. **视觉是眼睛，M0 是大脑**：K230 负责所有视觉感知任务，输出结构化数据给 M0，M0 根据状态机做出控制决策。
2. **多传感器冗余**：超声波、编码器、灰度传感器要保留作为视觉的补充，关键决策（如停车、路口）不能仅依赖单一传感器。
3. **状态机驱动**：你们的 StateCore 框架非常适合扩展第二阶段的状态，保持原有架构，增加新 App 和新状态即可。
4. **先简后繁，逐步验证**：先跑通双线巡线（基础），再叠加路口转向，再加红绿灯，再加停车，不要一上来就全部集成。
5. **时间紧迫，聚焦核心**：如果某些功能太难（如完美侧方停车），可以先做简化版（垂直倒车），保证整体流程能跑完，再优化细节。
