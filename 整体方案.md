## 二、推荐分工方案：**1 电控 + 1 视觉 + 1 系统集成**

| 角色 | 负责人 | 技术方向 | 核心职责 |
|------|--------|----------|----------|
| **A：M0 电控工程师** | 1 人 | MSPM0G3507、FreeRTOS、控制算法 | 扩展状态机、实现所有控制逻辑、电机/传感器驱动 |
| **B：K230 视觉工程师** | 1 人 | Python/OpenMV、图像处理、检测算法 | 所有视觉检测模块开发、标定、调参 |
| **C：系统集成工程师** | 1 人 | 通讯协议、上位机、路径规划、联调 | M0↔K230 协议、蓝牙上位机、路径规划、计时、测试 |

> **为什么不让 2 人做视觉？**
> 视觉任务虽然多，但每个模块的**算法思路是相似的**（都是颜色分割 + 形状检测 + 坐标计算），一个人统一开发能保证风格一致、避免接口冲突。而且视觉调试需要**实际场地**，两个人抢摄像头反而低效。
> 电控任务虽然杂，但**一阶段框架已经成熟**，`StateCore`、`SpeedMixer`、`Application` 基类都是现成的，主要是写新 App 和调参，一个人足够。

## 三、各角色详细任务清单

### 🅰 M0 电控工程师（1 人）

**继承一阶段成果**：`StateCore`、`SpeedMixer`、`Application` 框架、电机驱动、超声波驱动。

#### 任务 1：新增 Mods 层 —— K230 通讯驱动
- 新建 `Mods/vision_k230.cpp/hpp`
- 实现 UART 接收中断 + 环形缓冲
- 帧解析函数：帧头 `0xAA` 同步 → 读帧类型 → 读数据段 → 校验和验证
- 数据缓存：将解析后的车道线偏移/角度、红绿灯状态、路口类型等存入结构体
- 发送指令函数：`SendCommand(uint8_t cmd)` 用于切换 K230 工作模式

#### 任务 2：新增 App —— 视觉巡线 (`VisionTrack.cpp/hpp`)
- 继承 `Application`，预分频 `prescaler = 1`（200Hz）
- `Update()` 中：
  - 从 `VisionK230` 读取 `offset`（mm）和 `angle`（°）
  - PD 控制器：`turn = Kp * offset + Kd * angle`
  - 输出到 `SpeedMixer`（新增 `VISION_TRACK` 来源，优先级低于 OVERTAKE、TURN_AROUND）
- 参数：`Kp = 0.015`, `Kd = 0.008`（需实测微调）
- 丢线处理：连续 5 帧 `valid == false` → 减速停车，状态置 `Warning`

#### 任务 3：新增 App —— 路口转向 (`IntersectionTurn.cpp/hpp`)
- 当 K230 上报路口检测（`intersection_type > 0`）时触发
- 根据**预设路径表**（由角色 C 定义）决定转向方向
- 转弯动作：原地旋转 90°（编码器辅助）
  - 左转：左轮反转，右轮正转，直到编码器累计脉冲达到 90° 对应的弧长
  - 右转：镜像操作
  - 转弯完成后清编码器计数，回到巡线
- 防误判：连续 3 帧检测到路口才触发（避免噪点）

#### 任务 4：新增 App —— 红绿灯响应 (`TrafficLight.cpp/hpp`)
- 当 K230 上报 `light_status == 1`（红灯）时：
  - `SpeedMixer` 清空所有速度，停车等待
  - 记录等待开始时间（扣除计时用）
- 持续查询 K230（每 200ms 一次），直到 `light_status == 2`（绿灯）
- 绿灯后恢复巡线，记录等待结束时间

#### 任务 5：新增 App —— 道闸通行 (`GatePass.cpp/hpp`)
- 接近停车场时（根据路径表到达 J 点），切换到道闸检测模式
- 向 K230 发送指令 `0x11`（切换为道闸检测模式）
- 如果 `gate_status == 1`（落下）→ 停车等待
- 如果 `gate_status == 2`（抬起）→ 通过，切换为停车位检测模式

#### 任务 6：新增 App —— 自主停车 (`Parking.cpp/hpp`)
- 进入停车场后，向 K230 发送指令 `0x12`（停车位检测模式）
- 子状态机：
  1. `SEARCH_SPOT`：低速巡行，等待 K230 上报空车位坐标
  2. `ALIGN_TO_SPOT`：根据车位坐标调整位置（车头对准车位入口）
  3. `BACK_IN`：倒车入库（编码器控制距离 + 超声波监测后方）
  4. `ADJUST`：微调（如有需要）
  5. `PARKED`：停车并标记完成
- 使用超声波监测后方/侧方距离，防止碰撞边界

#### 任务 7：编码器里程计完善
- 在 `motor_at8236` 或独立模块中，利用编码器脉冲累加计算左右轮位移
- 提供接口：`GetLeftDistance()`, `GetRightDistance()`, `ResetOdometry()`
- 用于路口转向角度计算、停车入库距离控制、位置估算

#### 任务 8：SpeedMixer 扩展
- 新增来源：`VISION_TRACK`（优先级 > TRACK，< OVERTAKE）
- 新增来源：`PARKING`（优先级最高，与 NAVIGATION 同级）
- 确保各种 App 的速度优先级正确仲裁

#### 任务 9：全局计时
- 在 `RobotSystem` 中维护：
  - `total_drive_time`：从出发到停车的总时间
  - `red_light_wait_time`：累计等红灯时间
- 最终通过 OLED 或蓝牙输出：`effective_time = total_drive_time - red_light_wait_time`

### 🅱 K230 视觉工程师（1 人）

**技术栈**：MicroPython / OpenMV IDE，K230 的 NPU 可选（分类任务可考虑，但简单任务先用传统 CV）

#### 任务 1：环境搭建 & 摄像头标定
- 烧录 MicroPython 固件到 K230
- 安装 OpenMV IDE，配置串口通讯（与 M0 通讯用 UART，调试用另一个 USB 口）
- **摄像头标定**：
  - 畸变校正：打印棋盘格，拍 20 张不同角度照片，用 OpenCV 离线计算内参矩阵 `K` 和畸变系数 `dist`，写入 K230 代码
  - 透视变换矩阵：在场地地面上放置四个标记点（构成矩形），手动记录图像坐标 → 世界坐标，计算单应性矩阵 `H`，写入代码

#### 任务 2：双线车道线检测
- 图像预处理：高斯滤波 → 转灰度 → Canny 边缘检测
- ROI 提取：取图像下半部分（近处车道线更清晰）
- 透视变换：应用矩阵 `H` 转为鸟瞰图
- 左右车道线点提取：
  - 对鸟瞰图按列扫描，找到每行最左和最右的边缘点
  - 分别归入左车道线集合和右车道线集合
- 二次曲线拟合：`x = a*y² + b*y + c`
- 计算：
  - **横向偏移** `offset = (c_left + c_right)/2`（映射到 mm）
  - **航向偏差** `angle = atan((b_left + b_right)/2)`（转换为 °）
- 输出：`offset`, `angle`, `valid`（如果左右线点数 < 阈值，则 `valid = 0`）
- 发送帧：`0xAA 0x01 <offset_float> <angle_float> <valid_uint8> <checksum>`

#### 任务 3：红绿灯识别
- RGB 转 HSV
- 红色阈值分割：`red = (0, 100, 20) ~ (10, 255, 255)` 和 `(170, 100, 20) ~ (180, 255, 255)`（红色在 HSV 中跨 0°）
- 绿色阈值分割：`green = (35, 50, 50) ~ (85, 255, 255)`
- 形态学处理：开运算去除噪点
- 判断：如果红色区域面积 > 200 像素 → 红灯；绿色区域面积 > 200 → 绿灯
- **多帧确认**：连续 5 帧红灯才上报（避免单帧误检）
- 发送帧：`0xAA 0x03 <light_status> <reserved> <checksum>`

#### 任务 4：路口检测
- 在鸟瞰图中，监测车道线的**间距变化**：
  - 正常直道：左右线间距稳定
  - 接近路口：左右线间距突然增大，或车道线突然中断
- 路口类型判断：
  - 十字路口：左右线都向两侧扩展，且前方出现横向车道线
  - T 字路口：一侧扩展，另一侧为边界
- 发送帧：`0xAA 0x02 <intersection_type> <direction_suggestion> <checksum>`

#### 任务 5：停车位检测
- 在停车场区域（地面颜色变化或进入指定区域后切换模式）
- 寻找白色矩形框：`img.find_rects(threshold=5000)`
- 过滤：面积在合理范围内（对应车位大小 0.4×0.6m），矩形长宽比接近车位比例
- 判断是否为空位：检查矩形框内是否有物体（可通过超声波辅助，或视觉检测框内灰度变化）
- 计算车位相对位置（相机坐标系下）：
  - `spot_x`, `spot_y`（框中心相对车头的位置，mm）
  - `spot_width`, `spot_height`
  - `spot_angle`（车位方向角，如果平行于车道则为 0°，垂直为 90°）
- 发送帧：`0xAA 0x04 <spot_x_int16> <spot_y_int16> <spot_w_int16> <spot_angle_int16> <checksum>`

#### 任务 6：道闸识别
- 道闸杆通常为红白相间或黄黑相间的横杆
- 检测方法：颜色分割（红色/黄色）→ 形状检测（细长矩形）→ 判断倾角
  - 倾角接近水平（0°~15°）：杆落下 → 不可通行
  - 倾角接近垂直（75°~90°）：杆抬起 → 可通行
- 发送帧：`0xAA 0x05 <gate_status> <reserved> <checksum>`

#### 任务 7：所有检测函数的统一封装
- 用 `mode` 变量控制当前检测模式（`LANE`, `LIGHT`, `INTERSECTION`, `PARKING`, `GATE`）
- 主循环中根据 `mode` 调用对应检测函数
- 监听 M0 发来的指令切换 `mode`

## 五、关键建议

1. **通讯协议尽早冻结**：这是 A 和 B 的接口，一旦定下来就不能轻易改，否则两边都要返工。C 在第一第二天就要拿出协议文档初稿。

2. **视觉先做巡线，再做其他**：双线巡线是所有功能的基础（车都走不直，其他免谈），必须优先跑通。

3. **停车入库最难点**：这是第二阶段最大的技术挑战，不要放到最后才做。建议 B 在跑通巡线后就开始研究停车位检测，A 在跑通路口转向后就开始写停车控制，预留充足的调试时间。

4. **灰度传感器留作保险**：如果视觉在某段路况失效（如强反光、阴影），一阶段的灰度巡线可以作为降级方案，确保起码能跑完全程。

5. **所有阈值参数集中管理**：建议新建 `Config.hpp`，把所有 PID 系数、速度值、距离阈值、帧数阈值等放在一起，方便联调时快速修改，避免反复编译各个文件。
